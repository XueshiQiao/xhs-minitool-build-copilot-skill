# 可行性判定

> 对齐小红书官方规范 v1.2.0（快照 2026-07）。判定口径以
> [capability-snapshot.md](capability-snapshot.md) 为准。

把用户的需求拆成能力项，逐条过下面的红线。**命中任何一条红线，就不能说「可以做」。**

---

## 一、七条硬红线（命中即不可行）

| # | 需求里出现这类词 | 结论 | 为什么 | 替代方案 |
|---|---|---|---|---|
| 1 | 联网、接口、API、云端、同步、实时、后台拉取、在线数据 | ❌ | 容器禁一切网络请求（`fetch` / `XHR` / `WebSocket` / `SSE`），也禁加载外部图片字体媒体 | 数据打包成包内 `.json` 或写死在 JS；跨设备同步无解，直说 |
| 2 | 导出、下载、保存文件、生成图片保存、导出 Excel/PDF | ❌ | `a[download]` 与 blob 下载被禁，剪贴板 API 也被禁，长按菜单同样禁用 | **只能引导用户截屏**。把结果排版成一屏能截完的样子 |
| 3 | 复制到剪贴板、一键复制 | ❌ | `navigator.clipboard` 与 `execCommand('copy')` 均被禁 | 把文本渲染成可选中的（别全局 `user-select:none`），引导手动选中 |
| 4 | AI、模型、抠图、识别、OCR、语音转写 | ❌ | 既不能联网调模型，也不能跑 WASM（Draco / ONNX / Basis 这类加速库全部不可用） | 换成纯 JS 能算的规则逻辑；重计算类直接劝退 |
| 5 | 定位、附近、地图、蓝牙、NFC、摇一摇、计步、陀螺仪 | ❌ | 定位、硬件连接、各类传感器、`DeviceMotion`/`DeviceOrientation` 全部禁用 | 摇一摇改成点按；地图类无解 |
| 6 | 提醒、通知、推送、闹钟、后台运行 | ❌ | 系统通知/推送不支持，Worker 全系（Web/Shared/Service）禁用，无后台执行 | 只能用户主动打开时展示；「今天还没打卡」这类提示只能在页内做 |
| 7 | 跳转、外链、打开网页、跳到小程序/其他小工具 | ❌ | `window.open`、`target="_blank"`、跳转站外、小工具互跳全禁 | 单页内 JS 切视图 |

## 二、三条黄线（可做，但必须先告知用户代价）

### 黄线 1 · 数据不保证留得住

官方文档原话是「**请勿假设数据永久持久化**」。而申请持久化保护的
`navigator.storage.persist()` **本身就在禁用清单里**——连请求保护的资格都没有。

叠加「禁下载 + 禁剪贴板 + 无网络」，用户**没有任何备份手段**。

- 换手机 → 没了
- 清理 App 缓存 → 可能没了
- 系统存储紧张时被回收 → 可能没了

**判定**：⚠️ 可做，但不能把工具定位成「长期数据资产」（记账、日记、健康档案这类要慎重）。
界面上必须如实告知数据只存在本机。把「值得截图的阶段性成果」做成主要出口。

### 黄线 2 · 选文件只能选到图片和视频

`<input type="file">` 被系统选择器接管，**无论 `accept` 写什么，都只能选图片和视频**。

**判定**：⚠️ 想让用户传 CSV / JSON / 文本文件的需求，直接不可行。
且即使做选图，也**必须挡住用户选进来的视频**（判 `file.type`），否则流程会崩。

### 黄线 3 · 读者手里没有素材

这条不是技术限制，是**分发场景决定的**，但杀伤力一样大。

小工具的主场景是「读者刷到笔记 → 点开 → 立刻用」。如果你的工具要求用户
**先准备素材再上传**（两张对比图、一份文档、一段特定数据），
读者当下没有，动机极弱——点开、看到「请上传」、退出。

**判定**：⚠️ 技术可做，但产品上大概率没人用。详见 [product-fit.md](product-fit.md)。

## 三、明确可用的能力（放心用）

| 能力 | 说明 |
|---|---|
| HTML / CSS / JS 全套 | Flexbox、Grid、动画、媒体查询、`conic-gradient` 等标准特性 |
| Canvas 2D | `getContext('2d')` 完整可用 |
| WebGL | 纯渲染可用，纹理必须来自包内资源或内存对象 |
| 本地存储 | `localStorage` / `sessionStorage` / `IndexedDB` / `Cookie` / `Cache API`，按小工具隔离（注意黄线 1） |
| 摄像头 / 麦克风 | `getUserMedia`，需用户手势触发 + 系统授权 |
| 选图 / 拍照 | `<input type="file">`（注意黄线 2） |
| 音视频播放 | `<video>` / `<audio>` 内联播放，媒体须打包在内 |
| 触摸与指针手势 | `touch*` / `pointer*` 事件 |
| 包内字体 | `@font-face` 引用包内 `.woff` / `.woff2` |
| 内联 SVG | 作为标记写在 HTML 里，不算外部请求 |
| `alert()` / `confirm()` | 可用（原生 UI）。但会打断操作，建议自建页内组件；`prompt()` 被禁 |

## 四、判定输出模板

判完按这个格式回给用户，**逐条给理由，不要只说「不支持」**：

```
## 可行性：⚠️ 可做但要改

### 能做的部分
- 本地记录与统计 —— localStorage 可用
- 图表展示 —— Canvas 2D 完整可用

### 触到红线的部分
- ❌ 「导出 Excel」 —— 容器禁文件下载与剪贴板，没有任何等价实现
  → 替代：把结果排成一屏，引导用户截屏

- ⚠️ 「数据长期保存」 —— 官方明确「请勿假设数据永久持久化」，
  且禁用了申请持久化保护的 API，用户也无备份手段
  → 替代：不主打长期资产，改主打「阶段性成果截图」

### 建议
把「导出报表」改成「生成一张可截屏的月度卡片」，
既绕开了限制，也更贴小红书的分享场景。
```

## 五、快速自查（给用户的一句话版）

一个点子能不能做，先问这五个问题，**有一个「是」就要停下来重新想**：

1. 它需要连网吗？
2. 它需要让用户把结果**带走**（下载 / 复制 / 分享到别处）吗？
3. 它需要在用户没打开时做事（提醒 / 后台）吗？
4. 它需要用户**先准备好素材**才能开始吗？
5. 它的价值需要**长期数据积累**才成立吗？
